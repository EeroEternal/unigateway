# UniGateway 优先场景与用户分析

这份文档不讨论“还能加什么功能”，而是先回答一个更重要的问题：**UniGateway 现在最应该优先服务谁，以及为什么。**

对当前阶段的 UniGateway 来说，最重要的不是功能面铺得更大，而是把少数几个高频、刚需、可快速验证价值的场景打透。

## 1. 为什么要先看场景，而不是先看功能

这类项目很容易进入“不断补能力”的节奏：再加一个 provider、再补一个管理接口、再做一个 UI、再做更多策略配置。但如果没有明确场景，功能会越来越多，产品价值反而会越来越模糊。

对于 UniGateway，这个判断尤其重要，因为它当前已经具备了几个很明确的产品信号：

- 单二进制
- 本地或轻量部署
- CLI + 配置文件 + JSON Admin API
- 多 provider 路由、fallback、限流、模型映射

这说明它天然更适合解决“轻量、统一、可控”的模型接入问题，而不是一开始就做成重型平台。

## 2. 场景优先级的判断标准

如果要决定什么最该先做，可以用下面三个问题来判断：

1. 这个问题是不是用户真实高频遇到的？
2. UniGateway 当前形态是不是天然适合解决它？
3. 用户是否可以在很短时间内感受到价值？

越符合这三个条件的场景，越应该优先。

## 3. 当前最值得优先服务的用户

### 3.1 个人开发者 / AI 重度使用者

这类用户通常同时使用多个模型和多个 AI 工具，例如 Cursor、Codex、Claude Code、ChatGPT Agent，或者自己写脚本调用 OpenAI / Anthropic / DeepSeek。

他们最痛的点不是“没有模型可用”，而是：

- provider 多，配置分散
- 工具之间接入方式不统一
- 切换模型或切换供应商成本高
- 某个 provider 出问题时没有缓冲层

UniGateway 对这类用户的价值非常直接：**一个统一入口，接多个模型和多个上游。**

### 3.2 小团队共享网关的维护者

这类用户不是要一个复杂平台，而是想给团队提供一个“可统一发 key、可基本控流、可做简单路由”的共享入口。

他们关心的是：

- 如何给团队成员统一接入方式
- 如何控制额度、QPS、并发
- 如何在多个 provider 之间切换或 fallback
- 如何避免每个人手里都散落多个上游 key

UniGateway 当前的 service、provider、api key、routing_strategy 设计，与这个场景高度匹配。

### 3.3 希望降低供应商绑定风险的应用开发者

这类用户已经有业务接入模型，但不想把应用逻辑写死在某一家厂商的 API 上。

他们真正需要的是一个“迁移缓冲层”：

- 业务只接 UniGateway 一次
- 下游模型名可以稳定
- 上游 provider 可以替换
- 可以通过 model mapping 和 routing 做兼容

这个场景虽然不一定天天被提起，但价值很高，因为它决定了 UniGateway 能否从“临时工具”走向“长期基础设施”。

## 4. 最值得优先打透的三个场景

## 场景 A：AI 编码工具统一出口

### 用户目标

让多个 AI 编码工具都通过同一个网关出口访问模型。

### 用户真正买单的价值

- 一个 base URL / 一套接入方式
- 可以在 OpenAI、Anthropic、DeepSeek、Groq 等之间切换
- 某个 provider 失败时自动 fallback
- 不需要让每个工具都重新适配多家厂商

### 为什么这是最高优先级

这是当前 UniGateway 最“天然契合”的场景：

- OpenAI / Anthropic 兼容接口已经具备
- CLI quickstart 很适合本地快速起步
- 单二进制降低了试用门槛
- 多 provider 和路由已经有基础能力

### 对产品的启发

如果优先服务这个场景，那么很多产品决策会变得清晰：

- onboarding 必须极简
- quickstart 必须足够顺滑
- provider 示例必须覆盖主流编码模型
- fallback 和模型映射比 UI 更优先

## 场景 B：小团队共享 LLM 网关

### 用户目标

团队内统一发 gateway key，用一个网关对接多个 provider。

### 用户真正买单的价值

- 团队接入方式统一
- 可以做简单额度和并发控制
- 可以按 service 划分不同用途
- 后端 provider 可替换，对成员透明

### 为什么这个场景值得优先

它和当前架构非常匹配，而且很容易形成真实使用：

- `api_keys` 已经支持 quota / qps / concurrency
- service/provider/binding 的抽象已经够用
- admin API 已经能满足基础管理

### 当前短板

如果要把这个场景真正打透，默认安全策略需要更谨慎：

- admin token 不能默认放开
- 文档里需要明确公网暴露的风险
- 配置文件明文存储 key 的边界要说清楚

也就是说，这个场景不是不能做，而是需要先补安全默认值。

## 场景 C：多模型切换与迁移缓冲层

### 用户目标

业务代码只接一个稳定接口，但底层 provider 可以换。

### 用户真正买单的价值

- 避免应用层直接绑定某一家厂商
- 降低切换模型的迁移成本
- 为成本优化和质量调优预留空间
- 为生产环境 fallback 提供基础设施层支持

### 为什么这个场景重要

这决定了 UniGateway 的长期价值上限。

如果只是“本地代理工具”，项目价值会停留在效率层；如果能稳定承担“供应商解耦层”，它就更像一个真正的基础设施组件。

### 对产品的启发

这个场景会推动以下能力优先级上升：

- model mapping 的稳定性和可理解性
- provider 选择策略的可预测性
- fallback 行为的明确边界
- 更清晰的配置和错误提示

## 5. 当前不建议优先做的场景

### 5.1 重 UI 管理平台

现在 UniGateway 最强的是“轻量和直接”，不是“完整控制台体验”。

过早投入 UI，容易把注意力从核心接入体验、路由能力、配置稳定性上拉走。

### 5.2 企业级治理全套能力

例如复杂 RBAC、多租户账单、审批流、审计平台集成。

这些方向当然有价值，但它们依赖于更成熟的产品定位和更稳定的底层能力，不适合作为当前优先项。

### 5.3 泛化 SDK 平台路线

当前项目的核心竞争力不在 SDK，而在统一网关本身。

在网关的核心场景尚未打透前，过早把重心放在 SDK，容易分散资源。

## 6. 从场景反推近期产品优先级

如果把上面的判断落实到近期迭代，建议优先级如下：

1. 把“个人开发者 / AI 编码工具统一出口”做成最顺滑的体验
2. 把“小团队共享网关”补齐最基本的安全与管理能力
3. 把“多 provider 切换 / fallback / model mapping”打磨成可预期、好理解的核心能力
4. 最后再考虑更重的 UI、企业治理、平台化延展

这意味着近期最值得投入的方向通常不是更多表层功能，而是：

- 更好的 quickstart 和示例
- 更明确的 provider 配置模板
- 更稳定的 fallback 语义
- 更可靠的默认安全行为
- 更清晰的错误信息和运维边界

## 7. 结论

UniGateway 当前最适合服务的，不是“所有需要 LLM 的用户”，而是：

- 想统一多个模型入口的个人开发者
- 想提供共享 LLM 出口的小团队
- 想降低供应商绑定风险的应用开发者

只要优先把这三类需求服务好，UniGateway 的产品方向就会非常清楚：它不是一个重平台，而是一个**轻量、统一、可切换、可控的模型接入层**。

从这个角度看，场景不是文档里的“案例”，而应该是决定功能取舍、默认行为和迭代顺序的第一原则。
